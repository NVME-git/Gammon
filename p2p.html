🌐 How Gammon P2P Works

Gammon uses browser-to-browser WebRTC data channels via PeerJS for multiplayer — no game server, no accounts, no install. One player's browser acts as host and source of truth; guests connect directly to it.

Player numbering: Players are numbered 1–4 in the game UI. Internally the code uses 0-based indices (playerIndex 0 = Player 1 / Host, 1 = Player 2 / first Guest, and so on). This page uses the human-friendly 1-based numbering throughout.

1. System Architecture

The host browser generates a random PeerJS peer ID which becomes the room code. Guests use the share link (?room=<peerId>) to open a WebRTC DataConnection directly to the host. PeerJS's free cloud servers handle only the initial ICE/STUN handshake — all game data flows peer-to-peer after that.

Guests have no direct connection to each other. When a guest sends a state update, the host rebroadcasts it to every other guest — the host is always the relay and arbiter.

Host browser (Player 1)

Runs new Peer() — PeerJS assigns a random UUID as the room ID. Listens for incoming connections. Owns the canonical game state. Rebroadcasts every state message it receives from a guest.

Guest browser (Player 2+)

Reads ?room=<id> from the URL and calls peer.connect(hostId). On connection open, immediately sends request_state so it can catch up to the current game.

3. Page Reload & Seat Reclaim

A player who reloads the page (or loses their connection) can rejoin mid-game and reclaim their original seat — the board picks up exactly where they left off.

sequenceDiagram participant G as 🟢 Player 2 (Guest) participant LS as 💾 localStorage participant H as 🟣 Player 1 (Host) Note over G: page reload — ?room= still in URL G ->> LS : read gammon_seat LS -->> G : roomId + playerIndex + timestamp alt TTL expired (over 2 hours) or wrong room G ->> H : request_state (no claimedIndex) H -->> G : assign — new playerIndex allocated else valid saved seat G ->> H : request_state + claimedIndex 1 alt seat is still free H -->> G : assign — playerIndex 1 restored Note over G: rejoins as Player 2, game resumes else seat already taken by someone else H -->> G : assign — new playerIndex allocated end end G ->> LS : write gammon_seat with new playerIndex + timestamp
The saved seat is stored in localStorage as { roomId, playerIndex, timestamp }. The host honours a claimedIndex only if that seat is not currently occupied by an active connection — protecting against two browsers claiming the same player slot.

The End Turn button exists to let a player review and optionally undo their last move before it is broadcast. The state is never sent until the player explicitly confirms — protecting against mis-clicks.

5. Extra Players (Spectators)

The host does not cap incoming connections at the game's player count. A fifth person can join a four-player Quadgammon game and the room link will still work. Here is what happens:

flowchart TD A["New connection from browser X"] --> B["Host assigns next available playerIndex\ne.g. index 4 for a 4-player game"] B --> C["assignGuest sends assign message\nwith full save and playerIndex 4"] C --> D["Guest receives assign — sets localPlayerIndex = 4"] D --> E{playerIndex less than numPlayers?} E -- yes --> F["Active player — Roll + Move enabled on their turn"] E -- no --> G["Spectator — board visible, Roll + Move locked every turn\ncurrentPlayer never equals localPlayerIndex"] G --> H["Spectator sees all moves in real time\nvia normal state broadcasts"]
Spectators receive every state broadcast and see the board update in real time. They simply can never become the active player because currentPlayer only cycles through indices 0 to numPlayers − 1. No special handling or UI message is shown — the board just stays locked.

6. Message Protocol

All messages are plain JSON strings sent over a reliable WebRTC DataConnection. Every message has a type discriminant.

type Direction Payload fields Purpose
assign Host → Guest playerIndex: number
save: GameSave		 Tells a guest which seat they occupy and hands them the full current game state. playerIndex is 0-based internally (0 = Player 1).
state Any → All save: GameSave Full game snapshot after a confirmed turn. Host rebroadcasts to all other peers.
waiting Host → Guest none Game not yet started — guest shows a waiting screen.
request_state Guest → Host claimedIndex?: number Sent immediately on connection open. Asks host for current state and optionally tries to reclaim a saved seat.
hit Any → All attacker: number
defender: number		 Triggers elimination animation on all peers when a checker is hit to the bar.

GameSave shape

{
  mode:          "bigammon" | "trigammon" | "quadgammon" | "unigammon",
  numPlayers:    2 | 3 | 4,
  players:       [ { name, color }, … ],          // players[0] = Player 1
  points:        number[][],   // points[playerIndex][pointIndex] = checkerCount
  bar:           number[],     // bar[playerIndex] = checkerCount on bar
  borneOff:      number[],     // borneOff[playerIndex] = count borne off
  currentPlayer: number,       // 0-based index of the player whose turn it is
  phase:         "rolling" | "moving",
  dice:          number[],
  usedDice:      boolean[],
  moveHistory:   MoveRecord[]
}

8. Key Implementation Notes

No server required after handshake

PeerJS's free cloud is only used for WebRTC ICE negotiation (exchanging candidate addresses). Once DataConnection.open fires, all traffic is direct browser-to-browser UDP/SCTP. If both peers are behind restrictive NATs, PeerJS falls back to its TURN relay automatically.

State-sync model: full snapshots, not deltas

Every state message carries the complete GameSave object — not a diff. This keeps the protocol simple and makes late-joining and reconnection trivial: the host just sends its current snapshot with no diff-tracking logic needed.

Turn enforcement

The UI layer in main.js checks game.currentPlayer === network.localPlayerIndex before processing any roll or move click. Guests physically cannot send a state update out of turn because the Roll button is disabled and click events are dropped when it is not their turn.

Checker-hit animation sync

When the active player's checker hits an opponent's blot, the active player's browser sends a separate hit message in addition to the normal state sync. This lets all peers play the pixel-art elimination animation in parallel without embedding animation state in the save object.
